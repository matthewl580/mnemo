# Contributing to Mnemo

Thank you for wanting to contribute to Mnemo.

Mnemo is a free, open-source, local-first study application founded and maintained by **@torstfugl** under the **@onemnemo** GitHub organization.

Please read this document before opening a pull request.

## Project license

Unless otherwise stated, contributions to Mnemo are made under the project license:

- Apache License 2.0

By submitting a contribution, you agree that your contribution may be used, modified, distributed, and maintained as part of Mnemo under the project license.

You retain copyright to your own contributions unless otherwise agreed in writing.

## Project governance

Mnemo uses a founder-led governance model.

The project owner and maintainers may accept, reject, modify, revert, or remove contributions. Contributing does not grant ownership, voting rights, release rights, administrative access, maintainership, or decision-making authority over the official project.

For more detail, see `GOVERNANCE.md`.

## No employment or payment promise

Contributions are voluntary.

Contributing to Mnemo does not create employment, contractor status, partnership, ownership, equity, revenue share, or a guarantee of future payment.

If Mnemo later receives donations, sponsorships, grants, or revenue from optional hosted convenience services, any funding distribution will be handled under a separate written policy.

## Developer Certificate of Origin

Mnemo uses the Developer Certificate of Origin, Version 1.1.

By contributing, you certify that you have the right to submit your work under the project license.

Contributors may be required to sign off commits using:

```bash
git commit -s -m "Describe your change"
```

This adds a line like:

```text
Signed-off-by: Your Name <your.email@example.com>
```

The sign-off means you certify the DCO statement in `DCO.md`.

## Before contributing

Please try to:

1. Read the `README.md`.
2. Read `coding-standard.md` if your change touches code.
3. Search existing issues and pull requests.
4. Open an issue first for large features, architecture changes, or major UI changes.
5. Keep pull requests focused and reasonably small.

## Pull request expectations

A good pull request should:

- explain what changed
- explain why the change is useful
- link related issues when relevant
- include screenshots or recordings for UI changes when possible
- include tests when practical
- avoid unrelated formatting or cleanup
- follow the existing architecture and coding style
- avoid adding large dependencies without discussion

## Architecture and dependency decisions

Mnemo values clean architecture, local-first behavior, modularity, performance, and long-term maintainability.

New dependencies should be intentional. A dependency may be rejected if it adds unnecessary complexity, size, licensing issues, security risk, platform issues, or architectural debt.

## AI-generated contributions

AI-assisted contributions are allowed, but contributors are responsible for the result.

Do not submit generated code, assets, or text unless you understand it, have reviewed it, and have the right to contribute it under the project license.

## Security issues

Please do not publicly disclose security vulnerabilities before giving the maintainers a reasonable chance to respond.

Use the contact instructions in `SECURITY.md`.

## Code of conduct

Participation in Mnemo is covered by `CODE_OF_CONDUCT.md`.

Be kind, be constructive, and remember that this is a passion project maintained by real people.
